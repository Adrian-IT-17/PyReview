01🧱What is OOP?
💡 Analogy

Think of a Car. Every car has the same blueprint — 4 wheels, engine, doors. But each actual car is different — different color, different owner. In OOP: blueprint = Class, each actual car = an Object (instance).

Object-Oriented Programming (OOP) organizes code by grouping related data (attributes) and behavior (methods) into reusable units called classes.

The 4 Pillars of OOP:

  • Encapsulation — Bundle data + methods together, hide internal details
  • Inheritance — A class can reuse and extend another class
  • Polymorphism — Same method name, different behavior per class
  • Abstraction — Expose only what's needed, hide complexity
Procedural (old way)OOP (new way)
Data and functions are separateData + functions live together in a class
dog_name = "Rex"
def bark(name): ...		class Dog:
  name = "Rex"
  def bark(self): ...
Harder to reuse and scaleReusable, organized, scalable
02🏗️Classes & Objects
💡 Analogy

A class is a cookie cutter. An object is the actual cookie. One cutter makes many cookies, each with different frosting (attributes).

Use the class keyword to define a class. Always use PascalCase for class names (e.g. Dog, BankAccount).

python
class Dog: def __init__(self, name, breed): self.name = name # instance attribute self.breed = breed def bark(self): print(f"{self.name} says: Woof!") # Creating objects (instances) dog1 = Dog("Rex", "German Shepherd") dog2 = Dog("Bella", "Poodle") dog1.bark() # Rex says: Woof! dog2.bark() # Bella says: Woof! print(dog1.name) # Rex print(dog2.breed) # Poodle
  • class Dog: — defines the class (blueprint)
  • dog1 = Dog("Rex", "German Shepherd") — creates an object (instance)
  • Each object has its own copy of attributes — dog1.name ≠ dog2.name
  • Both share the same bark() method defined in the class
03🔧__init__ and self
💡 Analogy

__init__ is a setup form that fills in automatically when you create an object — like a new phone setting itself up on first boot.

self means "my" — my name, my grade. It tells Python which specific object you're referring to.

__init__ (dunder init) is the constructor — runs automatically every time a new object is created. Every instance method must have self as its first parameter. Python passes it automatically — you never pass it manually.

python — __init__ and self
class Student: def __init__(self, name, grade): # Runs AUTOMATICALLY when you do Student(...) self.name = name self.grade = grade self.passed = grade >= 75 s = Student("Adrian", 88) # Python auto-calls: Student.__init__(s, "Adrian", 88) print(s.name) # Adrian print(s.grade) # 88 print(s.passed) # True
You writePython actually does
s.bark()Dog.bark(s) — s is passed as self automatically
self.nameThis specific object's name attribute
self.grade = gradeStore grade on THIS object only

05🧬Inheritance
💡 Analogy

A Dog is a type of Animal. Everything an Animal can do, a Dog can too — PLUS its own things. Animal = Parent class. Dog = Child class. Dog inherits everything from Animal and can add or override.

python — inheritance and super()
class Animal: # Parent class def __init__(self, name): self.name = name def eat(self): print(f"{self.name} is eating") def speak(self): print("Some sound") class Dog(Animal): # Child class — inherits from Animal def __init__(self, name, breed): super().__init__(name) # call Animal's __init__ self.breed = breed def speak(self): # OVERRIDE parent's speak() print(f"{self.name} says: Woof!") def fetch(self): # NEW method — Dog only print(f"{self.name} fetches the ball!") dog = Dog("Rex", "Husky") dog.eat() # Rex is eating (inherited ✓) dog.speak() # Rex says: Woof! (overridden ✓) dog.fetch() # Rex fetches ball! (Dog only ✓)
  • class Dog(Animal): — Dog inherits from Animal
  • super().__init__(name) — calls the parent's constructor
  • Dog automatically gets all of Animal's methods
  • Dog can override methods to change behavior
  • Python supports multiple inheritance: class C(A, B):
06🔒Encapsulation
💡 Analogy

Your phone has an internal battery counter. You can't directly set it to 200%. You can only interact through proper channels — charge it, use it. That's encapsulation: protect the data, expose only safe access points.

ConventionMeaningExample
namePublic — anyone can accessself.name
_nameProtected — by convention, don't access directlyself._salary
__namePrivate — Python enforces via name manglingself.__password
python — encapsulation with getters/setters
class BankAccount: def __init__(self, owner, balance): self.owner = owner self.__balance = balance # __ = private def get_balance(self): # GETTER — safe read return self.__balance def deposit(self, amount): # SETTER — safe write with validation if amount > 0: self.__balance += amount else: print("Invalid amount") acc = BankAccount("Adrian", 1000) print(acc.get_balance()) # 1000 ✓ # acc.__balance # AttributeError! Can't access directly ✗ acc.deposit(500) # Valid — balance becomes 1500 acc.deposit(-100) # Invalid amount — rejected ✓
07🎭Polymorphism
💡 Analogy

The speak() command means different things to different animals. Tell a Dog → Woof. Cat → Meow. Duck → Quack. Same method name, different result per class. That's polymorphism — "many forms."

python — polymorphism
class Dog: def speak(self): return "Woof!" class Cat: def speak(self): return "Meow!" class Duck: def speak(self): return "Quack!" # Same loop, different results — polymorphism in action animals = [Dog(), Cat(), Duck()] for animal in animals: print(animal.speak()) # Woof! # Meow! # Quack!
  • Same method name (speak), different behavior per class
  • The correct version runs based on which class the object belongs to
  • Method overriding — when a child class redefines a parent's method
  • Use super().method() to still call the parent version if needed
08✨Special Methods (Dunders)

Dunder = Double UNDERscore. Methods like __init__, __str__, __len__. Python calls them automatically in specific situations — you define them to customize how your objects work with built-in operations.

python — dunder methods
class Student: def __init__(self, name, grade): self.name = name self.grade = grade def __str__(self): # called by print() return f"Student: {self.name}, Grade: {self.grade}" def __repr__(self): # called in console / debugging return f"Student('{self.name}', {self.grade})" def __eq__(self, other): # called by == operator return self.name == other.name s1 = Student("Adrian", 90) s2 = Student("Maria", 85) s3 = Student("Adrian", 72) print(s1) # Student: Adrian, Grade: 90 print(s1 == s2) # False print(s1 == s3) # True (same name)
MethodWhen Python calls it
__init__Creating an object: Dog("Rex")
__str__print(obj) or str(obj)
__repr__Console / debugging output
__len__len(obj)
__eq__obj1 == obj2
__add__obj1 + obj2
✅ Quick Summary — Everything You Need
  • Class = blueprint  |  Object = instance built from the blueprint
  • __init__ = constructor (runs on creation)  |  self = current object reference
  • Instance attributes (self.x) = unique per object  |  Class attributes = shared by all
  • Inheritance: class Dog(Animal): — use super() to call parent
  • Encapsulation: __x = private, _x = protected convention
  • Polymorphism: same method name, different behavior per class
  • Dunder methods: __str__, __eq__, __len__ etc. customize built-in behavior
